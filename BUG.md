# Gradient Descent Bug: Zero-Magnitude Tangents Cause Degenerate Tie-Breaking

## Summary

`gradient_descent` almost always fails to descend toward the root/minimum because it
constructs tangent vectors with `magnitude = zeros(T, dim)`, which causes
`directional_exponent` to break ties in favour of the **constant** monomial,
making every directional derivative zero.

## Root Cause

`gradient_descent` selects a child polydisc by evaluating the directional derivative
at each child and taking the one with the largest norm:

```julia
# src/optimization/optimizers/gradient_descent.jl
tangents = [ValuationTangent(lower_point, collect(lower_point.center), zeros(T, dim(lower_point)))
            for lower_point in below_nodes]
grad_values = loss.grad(tangents)
val, ind = findmax([LinearAlgebra.norm(g) for g in grad_values])
```

The directional derivative of a polynomial at a tangent `v` is computed by
`directional_exponent`, which finds the monomial with minimum valuation-weight and
breaks ties via:

```julia
reduce((n, m) -> dot(n, v.magnitude) < dot(m, v.magnitude) ? n : m, ties)
```

When `v.magnitude = [0, …, 0]` (all zeros), `dot(n, magnitude) = 0` for **every**
monomial `n`, so the reduce always returns the **last** element of the tie list.
Tie lists are generated by `findall` over the weight array, so the last tied element
is typically the **constant** monomial `[0, …, 0]`.

Since the derivative formula is:

```
d_v = -sum(n) * |coeff(g, n)| * p^{-radius · n}
```

choosing `n = [0, …, 0]` gives `sum(n) = 0`, hence **`d_v = 0`** regardless of the
polynomial, the center, or the radius.

## When the Tie Occurs

For a degree-1 polynomial `f(x) = x − r` evaluated at a child disc of radius `ρ`,
expanding around the child center `c`:

| Monomial | Valuation weight |
|----------|-----------------|
| constant `c − r` | `v(c − r) + 0` |
| linear `x`       | `v(1) + ρ = ρ` |

The critical tie occurs exactly when `v(c − r) = ρ`, i.e. when the child disc
**straddles the root** — precisely the case where the derivative should be
informative. With `magnitude = 0` the constant term always wins the tie, the
derivative is zero, and `findmax([0, 0, …])` falls back to index 1, picking the
**first** child regardless of which one contains the root.

## Observed Effect

A sweep over random degree-1, 2, and 3 polynomials over Q₂, Q₃, Q₅ (precision 20,
50 steps, 30 trials each) showed:

| Field | Degree | Divergences |
|-------|--------|-------------|
| Q₂ | 1 | 26 / 30 |
| Q₂ | 2 | 24 / 30 |
| Q₂ | 3 | 20 / 30 |
| Q₃ | 1 | 27 / 30 |
| Q₃ | 2 | 27 / 30 |
| Q₃ | 3 | 28 / 30 |
| Q₅ | 1 | 30 / 30 |
| Q₅ | 2 | 29 / 30 |
| Q₅ | 3 | 29 / 30 |
| **Total** | | **240 / 270 (89%)** |

"Divergence" = gradient descent best loss is more than 100× worse than greedy descent
best loss on the same problem.  Typical failure mode: the optimizer drills straight
down along the "zero" branch (child with the same center as the parent) and never
reaches the root.

## Proposed Fix

Construct tangents with a **non-zero magnitude** that penalises the constant monomial
relative to the linear terms during tie-breaking.  One natural choice is to set the
magnitude component for the dimension that was shrunk to the child's radius:

```julia
tangents = map(below_nodes) do lower_point
    mag = collect(lower_point.radius) .- collect(param.radius)   # 1 in the shrunk dim, 0 elsewhere
    ValuationTangent(lower_point, collect(lower_point.center), mag)
end
```

With this magnitude, `dot([1], mag) = 1 > dot([0], mag) = 0`, so `reduce` picks
`[0]` only when it **strictly** dominates, and the linear term wins ties — giving a
non-zero derivative when the child straddles the root.
